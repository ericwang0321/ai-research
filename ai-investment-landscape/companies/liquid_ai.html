4业务模式与收费方式

Liquid AI 的商业模式是其最具差异化、也最值得一级市场理解的部分。它刻意回避了 per-token metering(按调用量计费)的云 API 模式,转而走「开源权重 + 商业 license 阈值 + OEM/on-prem 部署服务」三层结构。

4.1 模型授权:LFM Open License v1.0

  • License 名称:LFM Open License v1.0(自研许可,非 Apache 2.0,也非严格意义的 OSI 开源)。
  • 核心条款:模型权重免费开放(open weights),含商用权利,但设有年营收门槛——
  • 当使用方(企业/实体)年营收 < $10M 时,可免费商用;
  • 当年营收 ≥ $10M 时,必须另行与 Liquid AI 签署商业 license。
  • 严格性说明:因设有「营收门槛 + 使用规模条件」,该 license 在 OSI 严格定义下属于 open weights(开放权重)而非 open source(开放源代码)。这一点与 Meta Llama 系列的「Llama Community License(>700M MAU 需授权)」逻辑类似,但门槛维度不同(营收 vs. 用户数)。

来源:Liquid AI – LFM License、Liquid Docs – Model License、HuggingFace LICENSE 文件、Liquid AI Pricing

5产品与技术(LFM 架构创新)

5.1 学术根基:Liquid Neural Networks

LFM 的技术源头是创始团队在 MIT CSAIL 的 liquid time-constant networks(LTC) 与 closed-form continuous-time neural networks(CfC) 研究:

  • LTC(2021, AAAI):Liquid Time-constant Networks——Hasani、Lechner、Amini、Rus、Grosu。受线虫(C. elegans)神经系统启发,用连续时间微分方程定义神经元动态,网络规模可极小(曾用 19 个神经元实现无人机端到端驾驶)且具备因果性、可解释性、对扰动鲁棒的特性。
  • CfC(2022, Nature Machine Intelligence):Closed-form continuous-time neural networks——将 LTC 的 ODE 求解封闭化,大幅降低推理计算量,使连续时间网络可在毫秒级延迟下运行。
  • Hyena Hierarchy(2023):与 Stanford / Together AI 团队同期工作相关,提出用长卷积 + 门控(hyena operator)替代注意力机制,实现对长序列的亚二次复杂度建模。

LFM 即把上述三股学术脉络(LTC/CfC 的连续时间动态 + Hyena 的线性/卷积算子 + 现代 MoE 路由)工业化的产物。

来源:AAAI – Liquid Time-constant Networks、arXiv 2006.04439、Nature Machine Intelligence – CfC、MIT News 2022、Turing Post – LFMs & Hyena Edge

5.2 LFM 架构:非 Transformer 的混合体

LFM 不是单一架构,而是混合架构(hybrid),核心组件包括:

  1. Hyena / 卷积算子(attention-free):替代部分注意力层,处理长序列时复杂度为 O(N) 或 O(N log N) 而非 Transformer 的 O(N²)。
  2. 选择性注意力(selective attention):在关键层保留少量注意力,兼顾质量。
  3. Mixture of Experts(MoE):在 LFM-40B 及 LFM2.5-8B-A1B 中使用,实现稀疏激活——总参数大但激活参数小(LFM2.5-8B-A1B 总参 8B、激活约 1B,故命名 "A1B"),显著降低推理成本。
  4. 动力系统骨干(dynamical systems backbone):继承 LTC/CfC 的连续时间动态特性,使模型对长程依赖与扰动更具鲁棒性。

5.4 核心技术优势(官方宣称 + 第三方验证)

  1. 更小的内存占用:由于不用(或少用)注意力,LFM 的 KV cache 显著小于同规模 Transformer(Transformer 在长上下文下 KV cache 线性膨胀,LFM 因卷积/递归结构而受控)。官方称在 32K–128K 长上下文下,LFM-7B 的显存占用可低于 Llama-3-8B 数倍。
  2. 更快的推理:同等质量下吞吐更高、延迟更低(Shopify 案例中的 <20ms 端到端)。
  3. 长上下文原生支持:LFM2.5 支持 128K context,且因架构特性,长上下文的边际成本远低于 Transformer。
  4. 端侧可部署:LFM2.5-1.2B-Thinking 可在 <1GB 手机内存中运行推理模型;LFM2-350M 等可跑在 MCU 级硬件。

7单位经济性(端侧推理成本)

Liquid AI 的单位经济性逻辑与传统 LLM API 公司根本不同,理解这一点是评估其估值的关键。

7.2 商业单位经济性

  • 对客户:LFM 的价值主张是「同样的钱,更长的上下文 / 更低的延迟 / 更私有的部署」。对车企、电商、医疗等不能把数据发云的场景,LFM 的 on-prem 部署消除 per-token 云成本,把变动成本变为固定 license 费。
  • 对 Liquid:
  • 收入 = license 费(年订阅)+ OEM 项目费(一次性 + 维护)+ 部署优化服务费;
  • 边际成本主要为工程人力(模型适配、量化、硬件调优),无大规模 GPU 推理集群折旧(因为它不在云端跑 per-token 服务);
  • 毛利结构理论上高于云 API 厂商(后者需承担 GPU 折旧),但规模化受限于「能签多少 OEM/企业大单」。

7.3 与 AMD/Qualcomm 的协同经济学

AMD 领投的本质:Liquid 模型让 AMD 的消费级/边缘芯片(NPU、Ryzen AI、Instinct)能跑得动有竞争力的 LLM,从而提升 AMD 硬件的 AI 卖点。这是一种「软件补贴硬件」的反向飞轮——Liquid 的 license 收入对 AMD 是零头,但 AMD 多卖出的芯片价值远超投资额。同理 Samsung NEXT、Qualcomm 生态。

  • 云端 API 厂商(OpenAI/Anthropic/Cohere):非直接竞争(Liquid 不卖云 API),但在「客户预算争夺」上间接竞争。

8.3 竞争态势判断

LFM 的护城河不在「架构本身」(Hyena/Mamba 思路公开,任何人可复现),而在于:

  1. 创始团队对该架构的世界级研究深度(LTC/CfC 原作者,迭代速度快);
  2. OEM/芯片生态的排他性绑定(AMD 领投、奔驰多年期、G42 中东渠道);
  3. 数据 + 部署 know-how 的积累(端侧量化、硬件适配是脏活累活,先发优势明显)。

最大风险:Transformer 阵营通过工程优化(KV cache 压缩、量化、推测解码、MoE)持续缩小效率差距,使「非 Transformer」的差异化窗口收窄。

10主要风险

  1. 架构差异化窗口收窄(最大风险):Transformer 阵营通过 KV cache 压缩、量化(GPTQ/AWQ)、推测解码(speculative decoding)、MoE、ring attention 等工程手段持续提升长上下文与端侧效率。一旦差距 <2x,「非 Transformer」的采用理由大幅减弱。
  2. 生态惯性:Hugging Face、PyTorch、vLLM、TensorRT-LLM 等整个推理栈围绕 Transformer 优化;LFM 需维护自己的 kernel 与推理后端,工程负担重、第三方支持滞后。
  3. 收入兑现不确定:OEM/企业 license 模式签约周期长(车企 18–36 个月)、单笔大但波动高;GetLatka 估算的 $13.2M ARR 相对 $2B+ 估值,估值/收入倍数极高(>150x),依赖未来兑现。
  4. 人才稀释风险:学者型创始团队(Daniela Rus 仍为 MIT CSAIL 主任)在学术与管理双重负荷下,公司运营深度依赖少数核心人员。
  5. 地缘与监管:G42 合作涉及中东/主权 AI,若美中/美-中东科技管制升级,可能影响合作落地;同时欧盟 AI Act、美国出口管制对端侧模型分发的影响尚不明朗。
  6. License 合规风险:LFM Open License 的「$10M 营收门槛」法律可执行性与社区接受度参差,部分开发者可能因「非纯开源」而倾向 Llama / Qwen / Mistral(Apache)。
  7. 大厂直接下场:若 Google(MobileLM)、Apple(端侧 Foundation Model)、Meta(Llama Edge)在小模型端侧认真投入,Liquid 的细分市场被挤压。
  8. 「300+ variants」与部分运营数据未经官方证实:可能导致对该公司的规模认知偏差。
