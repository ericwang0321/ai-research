Anyscale

Anyscale 是 Ray 开源分布式计算框架背后的商业公司。它不是普通云主机或单一推理 API，而是帮助企业和 AI 团队在 GPU/CPU 集群上开发、扩展、部署和运维 AI workloads 的托管平台。

1一句话结论

Anyscale 是 Ray 开源分布式计算框架背后的商业公司。它不是普通云主机或单一推理 API，而是帮助企业和 AI 团队在 GPU/CPU 集群上开发、扩展、部署和运维 AI workloads 的托管平台。

模型训练、fine-tuning、batch inference、online serving、RL、data processing、simulation 和 Agent workloads 都需要跨机器调度、弹性扩缩、成本优化和生产运维。Anyscale 通过 managed Ray、workspaces、jobs、services、observability、autoscaling、BYOC 和企业支持收费。投资重点是它能否把 Ray 的开源心智转化为稳定云收入。

5产品与技术

5.1 Ray open-source

Ray 是分布式 Python 框架,核心抽象包括 tasks、actors、objects、placement groups 和 distributed scheduling。它让开发者用 Python decorator 和 actor model 将单机函数扩展到集群。Ray 生态包括:

  • Ray Core: tasks、actors、distributed object store;
  • Ray Data: scalable data processing for AI;
  • Ray Train: distributed training;
  • Ray Tune: hyperparameter tuning;
  • Ray Serve: scalable model serving;
  • RLlib: reinforcement learning;
  • KubeRay: Kubernetes operator。

Ray 的优势是通用性。Spark 更偏数据批处理,Kubernetes 更偏容器编排,HPC frameworks 更偏专业计算。Ray 试图成为 AI/ML workload 的 Python-native distributed runtime。

5.2 Anyscale Platform

Anyscale Platform 包括:

  • Workspaces: 开发环境,让工程师在云端/集群上写、测、调 Ray 应用;
  • Jobs: submit and run batch tasks;
  • Services: deploy long-running Ray Serve applications;
  • Clusters: autoscaling compute clusters;
  • Observability: logs、metrics、traces、Ray dashboard、cost monitoring;
  • Images and dependencies: 管理 runtime environments;
  • Security/admin: users、teams、cloud credentials、private networking;
  • Integrations: AWS、GCP、Azure、Kubernetes/KubeRay。

Anyscale 的产品目标是隐藏分布式系统运维复杂度,使 AI engineer 能聚焦模型和应用。

5.3 Ray Serve / inference

Ray Serve 是构建 online inference services 的关键组件,支持 Python-native deployments、autoscaling、model composition、request batching、multi-model serving。Anyscale 在 AI infra 竞争中的一个核心问题是: Ray Serve 是否能成为企业自建模型 serving 的稳定选择,与 BentoML、KServe、Seldon、Nvidia Triton、vLLM、TGI、Baseten、Fireworks、Together 等竞争或互补。

对于客户,Anyscale 的好处是 serving 与 training/batch/data/RL 都在同一 Ray runtime 内;缺点是如果只想快速调用 open model endpoint,Fireworks/Together/Baseten 这类推理云更简单。

5.4 Ray Data / batch inference

AI 应用常有大规模离线任务: 文档 embedding、视频/图像处理、synthetic data generation、evaluation runs、batch LLM inference、data cleaning。Ray Data 和 Anyscale jobs 适合这类任务。相比 Spark,Ray Data 在 Python UDF、GPU batch processing 和 ML pipeline 上更贴近 AI engineer;相比 Modal,Anyscale 更强调 Ray ecosystem 和 enterprise scale。

5.5 RayTurbo / performance optimization

Anyscale 官网和博客强调 RayTurbo 等性能增强,目标是让 Ray workloads 在 Anyscale 上跑得更快、更省钱。商业上,performance optimization 是 managed open-source 公司收费的关键理由: 如果客户自己用开源 Ray + Kubernetes 运行成本高、调试难,Anyscale 可以用更好 autoscaling、scheduler tuning、observability、support 和 optimized runtime 收费。

5.6 安全和治理

企业使用 Anyscale 通常会关心:

  • 自有云账户/BYOC;
  • VPC/private networking;
  • IAM/SSO/RBAC;
  • secrets management;
  • audit logs;
  • image provenance and dependency control;
  • cost budgets;
  • data locality。

这些能力决定 Anyscale 是否能从 AI startup 工具变成 enterprise platform。

7单位经济性 / 关键财务逻辑

Anyscale 的单位经济性需要按 platform fee + cloud resources + support 来看:

  1. Platform fee 毛利较高: ACU/node-hour 平台费本质是软件层,毛利高于转售 compute。
  2. Compute 成本取决于部署模式: 若客户在自有云账户运行,BYOC 让 Anyscale 不承担全部 GPU 采购风险;若 Anyscale 代管或转售资源,则利用率和云折扣影响毛利。
  3. GPU workloads 放大收入和风险: H100 8x 节点价格很高,少数大客户 usage 能带来高收入,也会带来云成本、配额和峰值供给风险。
  4. Enterprise support 提高 ACV: 大型企业愿意为 Ray troubleshooting、architecture guidance、performance tuning 和 SLA 付费。
  5. 开源转化率是关键: Ray 使用广泛,但很多客户可自建 KubeRay 或在 Databricks/云平台中运行。Anyscale 必须通过降低运维成本和提高性能证明付费价值。
  6. Serving 毛利不同于 batch/jobs: Online serving 要求低延迟和高可用,需要 over-provisioning;batch jobs 更容易用 spot/弹性资源优化成本。

若 Anyscale 能成为企业 AI workloads 的默认运行平台,财务模型接近 usage-based cloud software;若只是 Ray support,估值倍数会低很多。

8.1 与 Fireworks / Together / Baseten 的关系

Fireworks、Together、Baseten 更偏"帮客户托管模型推理/训练/部署",开发者可直接调 endpoint。Anyscale 更偏"给客户一个运行 AI workloads 的分布式平台"。关系如下:

  • 如果客户只需要 Llama/Qwen/Mistral 推理 endpoint,Fireworks/Together/Baseten 更快;
  • 如果客户需要大规模 batch inference、custom pipeline、RL、simulation、data processing、multi-step workloads,Anyscale/Ray 更通用;
  • Baseten 也做 model deployment 和 autoscaling,与 Ray Serve/Anyscale Services 有直接竞争;
  • Fireworks/Together 可能在 open model serving 上比 Anyscale 更专注成本/延迟优化。

Anyscale 的防守点是 workload breadth 和 Ray ecosystem;进攻点是把 training、batch、serving、RL 统一在同一 runtime。

8.2 与 Databricks / Snowflake 的关系

Databricks 与 Anyscale 关系最复杂。Ray 与 Spark 都源自 Berkeley 系生态;Databricks 在数据/ML/AI platform 上拥有强企业入口。Databricks Mosaic AI、MLflow、Model Serving、Vector Search 和 lakehouse 数据治理能覆盖很多 AI workflow。Snowflake 也通过 Cortex AI、Snowpark、container services、Arctic/open models 进入 AI application platform。

Anyscale 的优势:

  • Python-native distributed runtime 更贴近 ML/AI engineer;
  • 适合 GPU-heavy、RL、simulation、custom serving、heterogeneous workloads;
  • 不要求所有数据先进入 lakehouse/warehouse。

Databricks/Snowflake 的优势:

  • 企业数据、权限、治理和采购入口;
  • SQL/BI/ETL/ML 统一平台;
  • 与现有数据工作流绑定。

长期看,客户可能在 Databricks/Snowflake 中管理数据和治理,在 Anyscale/Ray 上运行更复杂 GPU/AI workloads。也可能由数据平台把 AI workload orchestration 内嵌,削弱 Anyscale 的独立预算。

8.3 护城河

Anyscale 的护城河:

  • Ray 开源社区和 mindshare;
  • 通用 AI workload runtime,覆盖 train/data/serve/RL;
  • enterprise managed platform 和 support;
  • performance optimization / RayTurbo;
  • 多云/BYOC/KubeRay 兼容;
  • 大规模生产案例积累。

脆弱点:

  • Ray 可以自建;
  • 开源用户未必付费;
  • GPU inference 云更易用;
  • 数据平台/云厂商采购入口强;
  • 复杂平台销售周期长。

9团队与背景

Ion Stoica 是 UC Berkeley 计算机科学教授,也是 Databricks 共同创始人之一,其参与为 Anyscale 提供了开源基础设施商业化经验。Robert Nishihara、Philipp Moritz 等 Ray 相关研究者将 Berkeley RISELab 技术转为产品。Michael I. Jordan 等学术背景也增强了 AI/ML 分布式系统可信度。

团队优势是深度系统工程和开源生态,但商业化需要补齐 enterprise GTM、cloud operations、support 和行业解决方案。Databricks 的成功证明 Berkeley 系开源 infra 可以成长为大平台;但也意味着 Anyscale 的投资者会用相似标准衡量其能否从开源工具走向企业数据/AI 平台。

10主要风险

  1. 开源商业化风险: Ray 用户很多,但企业可以自建 KubeRay 或使用云厂商托管功能。
  2. 平台复杂度风险: 通用 workload runtime 范围广,产品/销售/支持复杂。
  3. 竞争预算挤压: Fireworks/Together/Baseten 抢 inference,Modal 抢 serverless jobs,Databricks/Snowflake 抢企业 AI platform。
  4. GPU 成本和供给风险: 若 Anyscale 代管 compute,云折扣、配额和利用率影响毛利。
  5. 客户集中风险: 大型 AI workload 客户收入高但议价强,usage 可能波动。
  6. 技术替代风险: Kubernetes-native AI platforms、vLLM/Triton/KServe、自研 infra 可能减少 Ray 需求。
  7. 销售周期风险: Enterprise AI infra 涉及安全、云账户、数据、模型、SRE 多团队审批。
  8. 估值风险: 早期独角兽估值需要匹配高 ARR 和高增长,公开数据尚不足以验证。
