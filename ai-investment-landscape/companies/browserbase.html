Browserbase

Browserbase 是给 AI Agent 提供真实浏览器环境的基础设施公司。它把网页搜索/抓取、登录态处理、代理网络、模型网关、运行时和可观测性打包成可调用的 browser automation infrastructure。

1一句话结论

Browserbase 是给 AI Agent 提供真实浏览器环境的基础设施公司。它把网页搜索/抓取、登录态处理、代理网络、模型网关、运行时和可观测性打包成可调用的 browser automation infrastructure。

它的核心判断是：很多企业任务无法只靠 API 完成，Agent 需要像人一样登录、点击、填表、下载、上传和跨站执行。Browserbase 同时经营开源 Stagehand SDK 和托管浏览器运行平台。投资重点是它能否从开发者工具扩展为高可靠的生产级 Agent 执行层。

Browserbase 的问题定义很清楚: 大量业务工作并没有可用 API,或者 API 权限、字段、成本、接入周期不适合 agent 动态执行。网页才是人类完成任务的统一界面。传统自动化工具 Playwright、Puppeteer、Selenium 能控制浏览器,但生产级 AI agent 还需要隔离 session、并发调度、proxy、CAPTCHA、身份、下载上传、回放、日志、模型选择、自然语言 action/extract、重试和企业治理。

因此 Browserbase 的商业化不是简单卖"云浏览器实例",而是卖一套 agent web execution stack: 先用 Stagehand 帮开发者把自然语言和 Playwright 控制结合起来,再把真实 browser sessions 托管在 Browserbase,同时给 Search/Fetch/Runtime/Identity/Observability/Model Gateway 打包。这个组合让它既像 Browserless/Sauce Labs 的浏览器基础设施,又像 Exa/Tavily 的 agent web API,还像 Modal/Daytona 的 agent runtime。

4.2 Browser session / compute economics

Browserbase 的最底层计费单位是 browser hour。真实浏览器比一次 API 调用更重: 需要 CPU、内存、网络、headless/headed 环境、隔离 profile、文件系统、录屏/回放、网络日志和清理。Developer 超额 $0.12/browser hr,Startup 超额 $0.10/browser hr,Scale usage-based,说明公司希望通过规模化和承诺用量把单价下降,同时用并发上限把免费滥用和突发成本控制住。

对客户而言,单位经济性取决于 agent 是否能把一次 web workflow 控制在较短 session 内。比如批量 KYC、供应商门户登录、招聘申请、价格监控、QA 测试、竞品抓取这类任务,如果每次都需要长时间加载、等待、重试或 CAPTCHA,费用会迅速增加。Browserbase 因此需要在 latency、session reuse、proxy quality、identity、Stagehand self-healing、Fetch/Search 替代浏览器打开等环节持续优化。

4.3 Search / Fetch / Extract 收费

Browserbase 不只让 agent 打开网页,也提供 Search API 和 Fetch API。Search 公开价为 $7/1k calls;Fetch 为 $1/1k calls,带 proxies $4/1k calls;Extract 为 $4/1k calls,带 proxies $7/1k calls。这与 Exa/Tavily 的 agent search/extract API 直接可比。逻辑是: 如果任务只是找到网页、读取内容、转成 markdown/JSON,客户不必启动完整 browser session;如果任务需要登录、点击、动态内容、上传下载或多步骤交互,才升级为 Browser API/Agent run。

这种组合能改善毛利: Search/Fetch 比浏览器会话轻,也能减少客户成本。Browserbase 如果能判断何时用 search/fetch,何时用 real browser,就能成为 web task router,而不是始终用最贵路径执行。

4.4 Agent runs、Runtime、Model Gateway

2026 年公司推出 Browserbase Agents,定位为"single API to extract structured data and interact with the whole web"。Agent run 把 prompt、model、browser、fetch/search、runtime、identity 和 output schema 组合成更高层的执行单元。Developer/Startup 各自包含 15/50 Agent runs,Scale 为 custom。

Model Gateway 按 market price pay-as-you-go,意味着 Browserbase 不试图把 LLM 成本藏进固定价,而是把模型调用作为 pass-through 或加价服务。它的价值在于用户可以用同一 API key 切换模型,并在 Stagehand/agent workflows 中对比模型在实际网页任务上的表现。官方 pricing 页也明确写到 Stagehand 与 Model Gateway 合起来可让团队 benchmark models against their actual web workflows and switch without code changes。

Runtime 则是把 browser agents 部署到 sandboxed environments,可以 on demand 或 scheduled 执行。它让 Browserbase 从"session provider"延伸到"agent deployment/runtime provider",与 Modal、Daytona、Cloudflare Workers、LangSmith Deployment、OpenAI Codex-style runtime 等存在边界重叠。

4.5 Identity、proxy、CAPTCHA 与 enterprise contract

Identity 是 Browserbase 收费与风险最复杂的部分。官网和定价页强调 auto CAPTCHA solving、Verified Agents、login/auth flows、proxy network、browser identity。Developer 包含 1GB proxies,超额 $12/GB;Startup 包含 5GB,超额 $10/GB;Scale 是 usage-based,Identity 从 auto CAPTCHA solving 升级到 Verified Agents + CAPTCHA solving。

企业合同的关键很可能包括:

  • 更高 concurrent browsers 和 browser hours commitment;
  • 更长 data retention、session replay、logs 和审计;
  • private networking、SSO/RBAC、DPA、security review;
  • proxy/identity SLA 和合规条款;
  • dedicated support/Slack、solution engineering;
  • 自定义模型网关、数据保留和日志导出;
  • 对高风险行业如 healthcare、tax、legal、supply chain 的条款约束。

Browserbase 的企业采购逻辑和传统 SaaS 不同: 客户不只是买工具,还把自己的 agent 行为、目标网站、登录状态、表单数据、下载文件和自动化策略交给第三方执行。因此安全、数据隔离、凭证管理、反滥用和合规条款直接决定能否进入 enterprise。

5产品与技术

5.1 Browser API / Sessions

Browser API 是核心原语,让开发者创建、控制、观察 browser sessions。官方文档写到 Browserbase 提供 cloud browsers、web search、page fetching、sandbox runtime 和 major LLM access。它兼容 Playwright、Puppeteer、Selenium,这对开发者采用很重要: Browserbase 不要求客户完全换框架,而是把现有 browser automation 迁到托管浏览器。

技术上,生产 browser sessions 要解决:

  • session isolation: 每个 agent/workflow 的 cookies、local storage、downloads、network state 隔离;
  • concurrency: 同时启动几十到几百个浏览器,避免冷启动和资源争抢;
  • replay/debug: 记录 DOM、network、console、screenshots/video、actions;
  • file IO: 支持上传、下载、处理记录;
  • auth: 登录态、MFA、cookie banner、CAPTCHA、bot detection;
  • deterministic control: Playwright/Puppeteer 需要等待、selector、retry;
  • AI control: Stagehand 需要 act/extract/observe 等自然语言 primitive。

5.2 Stagehand

Stagehand 是 Browserbase 维护的开源 AI browser automation framework。官方 pricing FAQ 将其描述为把 Playwright-level control 与 AI primitives 结合,包含 act、extract、observe,并与 LangChain、CrewAI、Mastra 等 agent frameworks 连接。Stagehand 是 Browserbase 的 developer wedge: 开源框架让开发者先在本地或简单场景中使用,当任务需要规模、并发、proxy、CAPTCHA、回放和生产运行时,自然迁移到 Browserbase。

Stagehand 的价值在于减少传统 selector 维护。网页 DOM 经常变化,按钮文案、布局、弹窗、cookie banner、A/B test 都可能让硬编码脚本失效。AI primitive 让开发者用"点击登录按钮"、"提取表格中的价格"这样的语义动作,由模型和页面观察去找元素。代价是推理成本、可解释性和稳定性: model 可能找错元素,所以 Stagehand 需要与 Braintrust/自家 observability/evals 结合,形成可靠性测试。

5.3 Browserbase Agents

Browserbase Agents 是 2026 年 6 月底推出的更高层产品,目标是让用户用一个 API 既提取结构化数据,又与整个 web 交互。它把 Search、Fetch、Browser API、Runtime、Model Gateway 和 Identity 组合到 agent run 中。对商业化重要的是,Agent runs 比 browser hours 更贴近业务结果,未来可以按 task/run、成功率、复杂度、行业模板和 enterprise workflow 收费。

这也使 Browserbase 进入更高层竞争: 如果客户只是要"做某个网页任务",不一定关心底层 browser session。Browserbase 可以从 infra 升级为 task automation platform,但同时会与 Zapier AI Agents、Airtable Hyperagent、Clay、n8n、Apify actors、UiPath agentic automation、OpenAI Operator-like 产品竞争。

5.4 Search API / Fetch API

Search API 面向 agents 提供快速、token-efficient web search results;Fetch API 将 URL 转成 clean markdown/HTML/JSON,可经 proxy 网络抓取。官网 use cases 包括监控 Google trending keywords、为 Claude Code 提供 live web access、验证 business registries for KYC 等。

这两个 API 的战略意义是"不要每件事都开浏览器"。很多 agent web tasks 实际上是 read-only retrieval,Search/Fetch 成本更低、速度更快、稳定性更好。Browserbase 把它们和 Browser API 放在同一平台,让客户按任务选择执行路径。

5.5 Runtime、Identity、Observability、MCP

Runtime 支持把 browser agents 部署、排程、按需运行。Identity 让 agent 通过登录、CAPTCHA、verified agent 等机制像人类一样导航。Observability 提供 replays、logs、prompts 的统一调试。MCP 支持把 browsing skills 暴露给 Claude Code、IDE agents 或其他 MCP clients。

这些模块组成了 agent ops layer。对于生产 agent,一次失败不只是"请求报错",而是可能卡在登录、弹窗、下载、站点改版、模型误点、网络异常、目标网站阻止自动化、输出 schema 不匹配。Browserbase 如果能把 browser session replay 与 model traces 结合,就能把自身从 commodity browser hosting 拉到 AI observability/runtime 更高价值层。

运营质量要看三组指标:

  1. usage-to-revenue conversion: 36.9M unique sessions 是否产生 browser-hour 收入,还是包含大量免费/测试/短 session。
  2. retention by workflow criticality: KYC、测试、数据抽取、内部 agent workflow 一旦上线,切换成本可能高;纯 search/fetch 任务切换成本低。
  3. cost per successful task: 如果 Browserbase 能把 session、proxy、model tokens 和重试次数降下来,客户会从 API scraping/自建 Playwright 迁移;反之会被成本压制。

公开资料中没有可直接回源的 ARR、收入 run-rate、gross margin、net retention、付费客户数、enterprise ACV 或估值。本稿不把第三方数据库中未能公开核验的数字写成事实。后续应向公司或投资人核验: 付费客户数、top 20 客户占比、browser-hour 毛利、proxy 成本、session success rate、Agent run 付费转化、Scale plan 合同规模。

7单位经济性 / 关键财务逻辑

Browserbase 的单位经济性可以拆成六层:

  1. 浏览器计算成本: 每个 session 消耗 CPU、内存、存储、网络和录制成本。browser hour 定价必须覆盖云资源、调度、隔离、重试和支持。
  2. proxy/identity 成本: 代理网络、CAPTCHA、登录、反自动化对抗是成本和风险高点。proxy GB 单独计费说明公司不会把这部分成本完全吸收在订阅内。
  3. LLM token pass-through: Model Gateway 按 market price,降低模型成本波动对毛利的冲击。若未来对模型调用加价,需证明选择、路由、eval、观测价值。
  4. Search/Fetch substitution: 能用 lightweight API 完成的任务不应启动完整浏览器。Browserbase 若能把 task routing 做好,客户 cost per task 降低,平台毛利改善。
  5. Enterprise support load: 高价值客户往往有复杂登录、代理、合规、站点适配和成功率要求,需要 solution engineering。毛利不仅取决于 compute,还取决于人工支持。
  6. Success-rate pricing potential: Browserbase 当前公开价按资源/调用收费,未来若 Agent runs 能稳定交付业务结果,可能按 successful run、workflow package 或 vertical template 提价。

关键财务逻辑不是"浏览器小时越多越好",而是"每个 workflow 的成功率和留存能否支撑浏览器成本"。如果 agent 需要反复重试、长时间等待和人工介入,客户会觉得每个成功结果过贵;如果 Stagehand/Agents/Fetch 能把任务压缩成短链路,Browserbase 可以形成高使用频次。

8.1 与 Exa/Tavily

Exa/Tavily 更偏 agent web search/retrieval,主要让 agent 找到和读取网页。Browserbase 则处理 Search/Fetch 之外的"网页需要像人一样操作"任务。边界在于 Browserbase 已加入 Search/Fetch,Exa/Tavily 也可能加入 crawl/actions。若客户只是实时搜索和内容抽取,Exa/Tavily 的 index/quality/latency 可能更强;若客户要登录、点击、填表和下载,Browserbase 更有必要。

8.2 与 Playwright/Puppeteer 自建

开源 Playwright/Puppeteer 免费且成熟。Browserbase 的护城河不是控制浏览器本身,而是生产化:

  • 并发和资源调度;
  • session replay/observability;
  • proxy/CAPTCHA/identity;
  • Stagehand AI primitives;
  • Model Gateway 和 eval;
  • enterprise security/retention/support;
  • agent templates 和 runtime。

只要客户规模不大,自建成本低;一旦浏览器任务进入生产并要求高并发和高成功率,托管平台的价值上升。

8.3 与 RPA

UiPath、Power Automate 等 RPA 更适合稳定企业内部系统和 desktop/process automation。Browserbase 更适合 web-native、动态、跨站、AI-driven 的工作流。长期两者会碰撞: RPA 厂商会把 LLM/agent 加进产品,Browserbase 也会进入企业流程自动化。Browserbase 的优势是 developer-first 和 web/agent 原生;RPA 优势是企业采购、合规、流程治理和集成生态。

9团队与背景

公开客户案例中 Paul Klein IV 以 Browserbase founder & CEO 身份出现。公司官网和博客展示多个工程、产品和社区作者,说明团队不只是做托管浏览器,也在持续输出 Stagehand、CAPTCHA/identity、browser agent benchmarks、conference/community、customer engineering 等内容。

团队判断重点:

  • 创始团队是否具备 browser infra、distributed systems、anti-bot/proxy、security、developer tools 背景;
  • 是否能同时服务开源社区和企业客户;
  • 是否能处理高风险网页自动化场景的合规边界;
  • 是否能把客户 workflow 成功率工程化,而不仅是卖算力;
  • 是否有足够 GTM 能力把 developer usage 转成 Scale 合同。

公开资料暂时不足以完整拆解管理层、员工数、组织架构和融资后团队扩张节奏。后续应核验 LinkedIn headcount、工程/销售比例、关键高管加入、客户成功团队规模。

10主要风险

  1. 可靠性风险: 网页随时变化,弹窗、A/B test、cookie banner、MFA、CAPTCHA 和反 bot 都会打断 agent。成功率不足会直接影响客户 ROI。
  2. 成本风险: 真实浏览器比 search/fetch/API 调用更贵。若每个任务 session 长、重试多、模型 token 多,客户可能回到 API 或人工。
  3. 合规与 ToS 风险: 自动登录、抓取、表单提交和代理网络涉及网站条款、数据授权、隐私和滥用边界。
  4. 安全风险: Browserbase 可能处理登录凭证、cookie、文件、客户数据和目标网站信息,必须防止泄露和越权访问。
  5. 平台替代风险: OpenAI/Anthropic/Google 若内置 browsing/operator/runtime,或云厂商/RPA 厂商提供企业级 agent browsing,会压缩独立平台空间。
  6. 开源替代风险: Playwright、Puppeteer、browser-use、Stagehand-like tools 让技术门槛降低,大客户可自建。
  7. 融资信息透明度不足: 缺少公开融资/ARR/估值,投资判断难以量化价格。
